# ROBO-Path 씬 덤프 도구 명세 (Scene Dump Tool Specification)

## 1. 목적
- 맵 제작은 시각적 작업이라, AI는 씬을 텍스트로만 인지할 수 있다.
- 씬 덤프는 씬의 구성 요소(거점, 타일, 장애물, 인접 관계)를 JSON으로 내보내, AI와 사람이 동일한 맵 상태를 공유하고 정밀하게 협업하기 위한 수단이다.
- 스크린샷은 첨부 제약이 크므로, 텍스트 덤프를 주력 수단으로 한다.
- 맵 설계 명세(`ROBO-Path_Map_Design_Specification.md`)의 태그 체계 및 데이터 모델과 연계된다.

## 2. 수집 범위 및 섹션 구조
- 씬의 모든 관련 오브젝트를 수집하되, 하나의 JSON 파일 안에서 목적별 섹션(테이블)으로 분리한다. AI는 목적에 따라 필요한 섹션만 읽어 정보 과부하를 피한다.
- 섹션 구성:
  - `summary`: 전체 통계 (맵 크기, 각 섹션 오브젝트 수, 고도 범위 등 한눈에 파악용)
  - `nodes`: 거점(`Node_Station`, `Node_Pickup`, `Node_Destination`) 및 `Waypoint`
  - `tiles`: 지형/경로/노면 타일 (`Terrain_*`, `Path_*`, `Road_*`)
  - `obstacles`: 장애물 (`Building`, `Obstacle`, `Prop_Pole`, `Prop_Tree`)
  - `adjacency`: 타일 간 인접 관계 (그래프 연결성)

## 3. 필드 스키마

### 3.1 모든 오브젝트 공통 필드
- `name`: 오브젝트 이름
- `tag`: 맵 설계 명세의 5계열 태그 중 하나
- `position`: `{ x, y, z }`
- `rotation_y`: y축 회전각 (도 단위, 대부분 y축만 의미 있음)
- `size`: `{ width, depth, height }` (바운딩 박스 기준)

### 3.2 거점(nodes 섹션) 추가 필드
- `usage`: `"station"` | `"pickup"` | `"destination"` | `null` (`Waypoint`는 `null`)

### 3.3 타일(tiles 섹션) 추가 필드
- `terrain_type`: 태그와 동일 값 (비용 프로파일 매핑용 명시)
- `elevation`: 블록 윗면(로봇 주행면)의 y 높이 (경사/계단 판단용)

## 4. 인접 관계 (adjacency 섹션)
- 덤프 시점에 타일 간 인접 관계를 계산하여 기록한다. (A* 경로 탐색이 노드 간 엣지를 필요로 하므로, 덤프만으로 그래프 구조를 파악할 수 있어야 함)
- 판정 방식: **변 맞닿음(4방향) + 허용 오차**
  - 두 타일이 인접하려면 한 축에서는 경계가 맞닿고(거리가 `tolerance` 이하), 다른 축에서는 실제로 면이 겹쳐야(overlap) 한다.
  - 대각선 인접(두 축 모두에서 맞닿기만 하여 꼭짓점만 공유하는 경우)은 의도치 않은 그래프 연결(버그)을 유발하므로 인접에서 엄격히 제외한다.
  - `tolerance` 기본값: `0.1` (미터). 부동소수점 좌표 연산으로 인한 미세한 틈을 보정하기 위한 허용 오차.
  - 겹침(overlap) 판정 시에도 `tolerance`보다 큰 길이만큼 겹쳐야 인정하여, 단순히 모서리 끝부분만 미세하게 닿는 경우를 배제한다.
- **권장 사항**: 맵 제작 시 타일을 격자(grid)에 스냅하여 배치하면 미세 틈 문제가 근본적으로 줄어든다.
- 기록 형식: 각 타일의 인접 타일 목록
  ```json
  { "tile": "이름", "adjacent_to": ["이름1", "이름2"] }
  ```

## 5. 저장 위치 / 파일명 / 실행 방식

### 5.1 저장 위치
- **저장 경로**: `Unity/ROBO-Path-Simulator/scene_snapshots/` (저장소 루트 기준 상대 경로)
- **사유**: 
  - 씬 덤프는 "설계 문서"가 아니라 Unity 시뮬레이터가 생성하는 데이터 산출물이므로 Unity 프로젝트 내부에 위치하는 것이 자연스럽다.
  - 에디터 스크립트가 자신의 프로젝트 폴더 기준으로 출력하므로 경로 계산이 단순하고 안정적이다.
  - 여전히 `Assets/` 외부이므로 Unity 에셋 임포트 대상이 아니며, 성능 저하를 방지할 수 있다.
  - 저장소에 상대 경로로 커밋되어 협업 주체(AI, 사람, Python 백엔드)가 쉽게 접근하여 읽을 수 있다.

### 5.2 파일명
- 항상 동일한 파일명으로 덮어쓴다: `scene_dump.json`
- 이력 관리는 타임스탬프 파일을 무수히 쌓지 않고 Git 히스토리를 활용한다.

### 5.3 실행 방식
- Unity 에디터 상단 메뉴에 버튼을 추가한다 (예: `ROBO-Path > Export Scene Dump`).
- 버튼 클릭 시 현재 열린 씬을 덤프하여 `5.1`의 경로에 저장한다.
- 구현은 에디터 전용 스크립트(`Assets/Scripts/Debug/` 하위, `Editor` 폴더 규칙 적용)로 예정됨을 명시한다.

## 6. 백엔드 데이터 모델과의 연계
- `nodes` 섹션의 거점 → `base_locations`(`location_usage`)로 매핑.
- `tiles` 섹션의 태그 → `config/cost_profiles.json`의 지형 프로파일과 연결되어 엣지 비용 산출에 사용.
- `adjacency` 섹션 → 향후 `map_edges`(노드 간 연결) 생성의 기초.
- 본 덤프는 씬과 백엔드 데이터 모델을 잇는 중간 표현(Intermediate Representation)임을 명시한다.
