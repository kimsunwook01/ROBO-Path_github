# ROBO-Path 사용자 인증 및 데이터 소유권 설계 (Auth Design)

**⚠️ 중요: 본 문서는 웹 공개를 대비한 향후 설계 방향을 정의합니다. 실제 인증 시스템(Streamlit UI, DB RLS 적용 등)의 구현은 Unity 시뮬레이터가 완성된 이후 단계에서 진행됩니다.**

---

## 1. 목적 및 범위
*   **사용 대상:** 프로젝트 관련 교수, 조교 및 허가된 연구 인력
*   **운영 정책:** 서버(맥미니 및 라즈베리파이)의 부하 관리를 위해 한 번에 한 명의 접속을 권장합니다.
*   **인증 제공자:** 사용자 계정 관리, 비밀번호 해싱 및 세션 유지는 **Supabase Auth**를 전적으로 활용합니다.

## 2. 인증 방식
*   **회원가입:** `아이디` + `비밀번호` + `인증키(가입 제한용)`
*   **로그인:** `아이디` + `비밀번호`
*   **권한 (Role):** 별도의 계급(관리자/일반사용자)을 구분하지 않습니다. 로그인에 성공한 모든 사용자는 운영자와 동일한 권한 등급을 가집니다.

## 3. 아이디 → 이메일 형식 변환 (내부 처리)
Supabase Auth는 기본적으로 사용자 식별자로 이메일 형식을 요구합니다. 사용자 편의를 위해 UI에서는 단순 `아이디`를 입력받고, 백엔드에서 이를 이메일 형식으로 변환하여 처리합니다.

*   **변환 규칙:** `{아이디}@robopath.app`
*   **특이사항:** `.local`과 같은 비표준 TLD는 Supabase 내부 유효성 검사에서 차단될 수 있으므로, 형식상 유효한 가상 도메인(`.app`)을 사용합니다. 실제 도메인을 구매하거나 등록할 필요는 없습니다.
*   **[필수 조치] 이메일 인증 비활성화:**
    우리는 가상 이메일을 사용하므로 가입 확인 메일을 수신할 수 없습니다. 따라서 Supabase 대시보드에서 다음 설정을 반드시 수동으로 변경해야 합니다.
    1. Supabase Dashboard 접속
    2. `Authentication` > `Providers` > `Email` 설정으로 이동
    3. **"Confirm email" (이메일 확인) 옵션 비활성화(Toggle Off)** 및 저장

## 4. 인증키 (가입 제한 로직)
외부 불특정 다수의 무단 가입을 방지하기 위해 '사전 공유 비밀번호(Pre-shared Key)' 형태의 인증키를 도입합니다.

*   **배포 방식:** 운영자가 사전에 고정된 인증키를 생성하여 교수진/관계자에게 별도로 전달합니다.
*   **보관:** 백엔드의 `.env` 파일에 보관합니다. (예: `SIGNUP_SECRET_KEY=roborobo2026`)
*   **검증 시점:** **회원가입 시점에만** 입력받아 검증하며, 로그인 시에는 요구하지 않습니다.
*   **검증 위치:** Streamlit 백엔드 서버에서 가입 폼 제출 시, 입력된 인증키가 `.env`의 값과 일치하는지 먼저 확인합니다. 일치할 경우에만 Supabase Auth의 Signup API를 호출합니다.

## 5. 데이터 소유권 모델 (Data Ownership Model)

데이터의 특성에 따라 '공용 데이터'와 '개인 데이터'로 소유권과 열람 범위를 분리합니다.

### 5.1 공용 데이터 (Public / Shared Data)
모든 로그인 사용자가 열람하고 갱신을 공유하는 집단 지성 데이터입니다.
*   **대상 테이블:** `nodes`, `base_locations`, `discovered_nodes`, `map_edges`, 로봇 정보, `platform_stats` 등
*   **사유:** 다양한 사용자가 여러 로봇을 조작하며 발견한 맵 탐색 결과(복셀, 경로 가중치)를 하나의 지도로 누적하여 "집단 공간지능"을 형성하는 것이 본 프로젝트의 핵심 목표이기 때문입니다.

### 5.2 개인 데이터 (Private Data)
개별 사용자가 발생시킨 이벤트성 데이터로, 본인의 것만 열람하고 제어할 수 있습니다.
*   **대상 테이블:** 
    *   `mission_logs` (사용자가 하달한 이동 명령 및 달성 여부)
    *   `incidents` (사용자가 입력한 자연어 피드백 및 수정 요청)
*   **소유권 식별:** 위 두 테이블에 `user_id` (UUID) 컬럼을 추가하여 소유자를 명확히 식별합니다.

## 6. RLS (Row Level Security) 정책 방향
*(본 섹션은 설계 방향만을 기술하며, 실제 SQL 적용은 구현 단계로 보류합니다.)*

*   **공용 테이블 정책:** 기존의 익명(anon) 읽기 허용 정책을 인증된 사용자(`authenticated`) 전용 읽기 허용으로 전환할 예정입니다.
*   **개인 테이블 정책 (`mission_logs`, `incidents`):** 
    사용자는 자신이 생성한 행(`auth.uid() = user_id`)에 대해서만 SELECT, INSERT, UPDATE 할 수 있도록 제한하는 RLS 정책을 수립합니다.
*   **시스템 우회 (Service Role):**
    시뮬레이터나 파이썬 백엔드가 백그라운드에서 데이터를 자동 적재하는 작업은, 기존과 동일하게 `service_role` 키를 사용하여 RLS를 우회하도록 유지합니다. 이를 통해 NULL 소유권으로 생성된 로그도 원활하게 적재될 수 있습니다.

## 7. 읽기 전용 모드와의 관계
*   **개념적 독립성:** "읽기 전용 모드(Read-only Mode)"는 인증/권한 계급과 무관합니다.
*   **활성화 조건:** 읽기 전용 모드는 오직 **시뮬레이터(맥미니) 오프라인 상태** 등 시스템 페일세이프(Failsafe) 상황에만 발동됩니다. 로그인 여부나 사용자 계정에 종속되지 않으므로 혼동하지 않아야 합니다.

## 8. 시뮬레이션 세션 관리 (Session Management)

### 8.1 세션 개념 정의
*   **게임 세이브 슬롯:** 세션(Session)은 독립된 탐색 상태를 담는 "게임 세이브 슬롯" 개념입니다.
*   **버전과의 차이:** 버전(Version) 관리는 한 세션 내부의 탐색 진행 차수를 의미하며, 세션은 완전히 독립된 시뮬레이션 환경을 의미합니다. (여러 개 공존 가능)
*   **데모 환경:** 데모나 시연 목적으로 아무것도 탐색되지 않은 백지 상태부터 체험할 수 있도록 세션 기능을 지원합니다.

### 8.2 세션 종속 vs 고정 데이터
*   **세션 종속 데이터 (스냅샷 보관):**
    *   `discovered_nodes` (발견 노드)
    *   `map_edges`의 `platform_stats` (피드백 통계)
    *   복셀 탐색 데이터
    *   `mission_logs` (미션 명령/결과)
    *   `incidents` (피드백 입력)
    *   `simulation_logs` (검증 로그)
*   **고정 데이터 (전역):**
    *   맵 구조: `base_locations` (BASE 노드), 길/지형
    *   `map_metadata`의 맵 구조 정의
    *   사용자 계정 (`auth.users`)
    *   로봇 기본 정의 (`robots`)
> **핵심:** 세션 종속 데이터는 활성 세션 동안 **현재 작업 테이블에 그대로 존재**하며, 세션 전환 시 스냅샷으로 묶여 라즈베리파이 SSD에 저장(`sessions.snapshot_url`로 참조)된 후 작업 테이블은 비워집니다. 반면 맵 구조는 세션 전환과 무관하게 영구적으로 고정 유지됩니다.

### 8.3 세션 동작 (스냅샷 아카이빙)
*   **세션 생성:** 작업 테이블을 초기화(비우기)하여 탐색 상태가 없는 새로운 백지 세션을 생성합니다.
*   **세션 전환 (스냅샷 저장 및 복원):**
    1.  현재 작업 테이블의 세션 종속 데이터를 직렬화하여 스냅샷 파일로 라즈베리파이 SSD에 저장합니다.
    2.  `sessions` 테이블에 현재 세션의 `snapshot_url`과 `last_saved_at`을 갱신합니다.
    3.  현재 작업 테이블의 세션 종속 데이터를 모두 비웁니다(초기화).
    4.  전환할 대상 세션이 기존 세션이면, 해당 `snapshot_url`에서 데이터를 읽어 작업 테이블로 복원합니다. 새 세션이면 빈 상태를 유지합니다.
    5.  대상 세션의 `is_active`를 TRUE로, 나머지를 FALSE로 전환합니다. (전체 사용자 전역 전환)
*   **세션 삭제:** 비활성 상태인 불필요한 세션(및 연결된 스냅샷 파일)을 제거합니다. (활성화된 세션은 삭제 불가)
*   **이전 세션 복귀:** 저장된 스냅샷에서 작업 테이블을 다시 적재하여 이전 상태를 그대로 복원합니다.

### 8.4 동시성 제어
*   **접속 상태 추적:** 탭 열림 = 로그인(활성), 탭 닫힘 = 로그아웃으로 간주합니다.
*   **구현 방식:** 브라우저 탭에서 주기적인 하트비트 핑을 전송하며, `beforeunload`/`visibilitychange` 이벤트를 병행하여 상태를 추적합니다.
*   **세션 변경 차단 규칙:**
    *   본인 외 다른 활성 접속자가 있을 경우 세션 변경이 원천 차단됩니다.
    *   **차단 시 경고:** "다른 사용자가 접속 중이라 세션을 변경할 수 없습니다. (비정상 종료된 경우 다시 접속 후 로그아웃 해주세요.)"
*   **단순성 유지:** 연구실 내 대면 사용을 전제로 하므로 과도한 타임아웃이나 복잡한 자동 정리 로직은 구현하지 않습니다. 수동 안내로 충분합니다.

### 8.5 세션 전환 시 안전 알림
*   세션 전환은 전체 사용자에게 영향을 주는 중대한 작업이므로, 실행 전 반드시 아래와 같은 확인 알림을 표시합니다.
    *   "세션을 전환하면 현재 작업이 저장되고 모든 사용자가 새 세션으로 전환됩니다. 계속하시겠습니까?"

### 8.6 Streamlit UI (향후 구현 항목)
*   세션 목록 표시 (현재 활성 세션 하이라이트)
*   세션 생성 / 전환 / 삭제 버튼
*   현재 접속자 수 표시
*   세션 변경 차단 시 팝업 경고 처리

## 9. 구현 단계 선행 처리 항목 (Implementation Prerequisites)

> 본 인증/세션 기능은 현재 설계 및 마이그레이션 준비만 완료된 상태이며, 실제 구현은 Unity 시뮬레이터 완성 후 진행한다. 구현에 착수할 때 아래 항목을 가장 먼저 처리해야 한다.

### 9.1 RLS 미적용 테이블 처리
*   현재 `sessions`, `active_sessions` 테이블이 DB에 생성되었으나 RLS(Row Level Security)가 비활성화되어 있어 Supabase에서 보안 경고가 발생한 상태이다.
*   **원인:** 마이그레이션 파일이 자동 배포 파이프라인에 의해 DB에 이미 적용되었기 때문 (아래 9.2 참조).
*   **처리:** 인증/세션 구현 시 아래 테이블에 RLS를 활성화하고 적절한 접근 정책을 적용한다.
    *   `sessions`
    *   `active_sessions`
    *   (그 외 `user_id`가 추가된 `mission_logs`, `incidents` 포함)
*   현 시점에서는 실제 서비스가 아니고 빈 테이블이므로 보안 경고를 그대로 두고 시뮬레이터 작업을 우선한다.

### 9.2 마이그레이션 자동 배포 충돌 주의
*   `.github/workflows/supabase-migrations.yml` 워크플로우는 `supabase/migrations/` 폴더에 변경이 생겨 main에 푸시되면 자동으로 `supabase db push`를 실행하여 폴더 내 모든 마이그레이션을 DB에 적용한다.
*   따라서 파일 상단의 "적용 보류" 주석은 자동 배포를 막지 못한다. (실제로 `user_id`, 세션 마이그레이션이 의도와 달리 이미 DB에 적용됨)
*   향후 "구현 시점까지 적용을 미루고 싶은" 마이그레이션이 있을 경우, `supabase/migrations/` 폴더에 두면 즉시 적용되므로 주의해야 한다. 진짜 보류가 필요하면 별도 폴더(예: `supabase/pending/`)로 분리하거나 구현 시점에만 `migrations/`로 이동하는 방식을 사용한다.
*   현재 이미 적용된 보류 마이그레이션(`user_id`, 세션)은 롤백하지 않고 그대로 두며, 구현 단계에서 RLS 정책 적용 및 실제 사용 로직과 함께 활성화한다.
