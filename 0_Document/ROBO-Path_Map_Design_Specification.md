# ROBO-Path 맵 설계 및 비용 프로파일 명세 (Map Design & Cost Profile Specification)

본 문서는 Unity 시뮬레이터의 가상 캠퍼스 맵 구성 요소, 태그 체계, 거점 구조, 노면 통행 규칙, 그리고 지형별 플랫폼 비용 프로파일을 정의한다. 이 명세는 씬(Scene) 구성, 씬 덤프 도구, 그리고 백엔드 데이터 모델(`nodes`, `base_locations`, `map_edges`)을 잇는 단일 기준점(Single Source of Truth) 역할을 한다.

---

## 1. 시나리오 및 맵 개요

### 1.1 시나리오: 캠퍼스 커피 배달 로봇
- 약 500m × 500m 규모의 가상 캠퍼스(연세대 미래캠퍼스 일부를 모델로 한 가상 지역. 실제 지명을 모델링하지 않음).
- 로봇은 평소 **충전 스테이션**에서 충전·대기한다.
- **배달 임무:** 주문 발생 시 → 카페(픽업 지점)로 이동하여 물건 수령 → 목적지로 배달 → 스테이션 복귀.
- **탐색 임무:** 카페를 경유하지 않고 스테이션에서 출발하여 자율 탐색 수행 → 스테이션 복귀.
- 주문은 시간의 흐름에 따라 사전 배치된 목적지 후보 중 하나에서 랜덤하게 발생한다(일반적인 배달은 정해진 주소로 주문되므로).

### 1.2 임무 흐름 (타일/노드 기준)
- 배달: `Node_Station` → `Node_Pickup` → `Node_Destination` → `Node_Station`
- 탐색: `Node_Station` → (자유 탐색) → `Node_Station`

### 1.3 수령·전달 트리거
- 별도의 트리거 볼륨을 두지 않고, **로봇이 특정 타일(노드) 위에 도달하면 이벤트가 발생**하는 방식으로 구현한다.
- 바닥이 개별 메쉬 타일의 집합이므로, 거점 타일에 도달 = 해당 이벤트(수령/전달/충전) 발생.
- 별도의 반경(trigger_radius) 개념은 두지 않는다. 타일 자체가 영역이다.

## 2. 바닥 구성 원칙: 메쉬 타일 모자이크

- 바닥은 단일 Unity Terrain을 사용하지 않고, **개별 메쉬 조각(블록)의 집합**으로 구성한다.
- **용어 정의:**
  - **블록(Block):** 바닥을 이루는 물리적 지형 단위(평지/경사/계단 등). 로봇이 그 윗면 위를 주행한다.
  - **타일(Tile):** 블록 위에 얹는 논리적/이벤트 영역(거점 등).
  - (본 문서 및 프로젝트 전반에서 영역 구분을 위해 종종 '타일'과 '블록'이 혼용될 수 있으나, 물리적 지형 단위는 블록, 그 위를 덮는 이벤트나 의미적 영역은 타일로 규정한다.)
- **블록 규격:**
  - 기본 블록 크기: `10m × 10m × 10m` 정육면체. footprint(바닥 면적)는 항상 `10×10`으로 고정하여 타일 간 인접 판정의 일관성을 보장한다.
  - 높이: `2m` 단위로 조절 가능(2, 4, 6, 8, 10 등).
  - 로봇이 주행하는 면은 블록의 윗면이다. (기준 배치: 블록의 윗면을 `y=0`에 정렬하도록 중심을 `y=-5` 등에 둔다.)
- 원칙: **영역 = 오브젝트 = 태그**가 1:1로 대응한다. 각 지형 블록과 타일은 독립된 오브젝트이며 고유한 태그를 가진다.
- 이유: 영역 구분이 명확하여 씬 덤프가 정밀해지고, AI 협업 시 좌표·종류 기준으로 정확히 합의할 수 있다. Terrain의 모호한 영역 구분 문제를 회피한다.
- 각 블록(타일)은 곧 경로 그래프의 한 영역(잠재적 노드/엣지)으로 직결된다.
- (참고) 블록 간 인접/연결 관계를 씬 덤프에 포함할지는 씬 덤프 명세에서 별도로 정한다.

## 3. 태그 체계 (5계열)

맵 구성 요소는 아래 5계열의 태그로 분류한다. 씬 덤프는 이 태그가 부여된 오브젝트를 수집 대상으로 삼는다.

### 3.1 거점 계열 (Node)
| 태그 | 역할 | DB 매핑 |
|---|---|---|
| `Node_Station` | 충전 스테이션. 모든 임무의 시작·종료. 보통 1개 | base_locations, location_usage='station' |
| `Node_Pickup` | 카페/픽업 지점. 배달 임무만 경유. 1개 이상 | base_locations, location_usage='pickup' |
| `Node_Destination` | 배달 목적지. 사전 배치, 다수 존재 | base_locations, location_usage='destination' |
| `Waypoint` | 순수 경로 노드(거점 아님). 교차로·갈림길 | nodes (node_type='BASE'), base_locations 미등록 또는 usage=null |

> 거점 3종은 모두 사람이 정의한 고신뢰 노드이므로 `nodes.node_type = 'BASE'`에 해당하며, `base_locations.location_usage` 필드로 station/pickup/destination을 구분한다. **기존 스키마 변경이 불필요하다.**

### 3.2 지형 계열 (Terrain)
| 태그 | 설명 |
|---|---|
| `Terrain_Flat` | 평지 |
| `Terrain_Slope` | 경사로·언덕 |

### 3.3 경로 계열 (Path)
| 태그 | 설명 |
|---|---|
| `Path_Stair` | 계단. **핵심 요소.** 바퀴 통행 불가, 보행 강점 구간 |
| `Path_Ramp` | 계단과 같은 고저차를 잇는 완만한 경사 우회로. 바퀴가 선호 |
| `Path_Tunnel` | 터널 (선택 구현) |

### 3.4 노면 계열 (Road)
| 태그 | 설명 |
|---|---|
| `Road_Sidewalk` | 인도. 로봇 기본 주행 영역 |
| `Road_Vehicle` | 차도. 고비용 회피 대상. 인도와 경계에 연석(높이차) 존재 |
| `Road_Crosswalk` | 횡단보도. 차도를 건너는 합리적 통로 |

### 3.5 장애물 계열 (Obstacle/Prop)
| 태그 | 설명 |
|---|---|
| `Building` | 건물 외관. 통행 불가(NavMesh 제외) |
| `Obstacle` | 벽·구조물 등 기타 장애물 |
| `Prop_Pole` | 가로등·전봇대·표지판 기둥. 가는 수직 장애물. Raycast 탐색에 걸림 |
| `Prop_Tree` | 나무. 수관 때문에 부피가 더 큼 |

## 4. 노면 통행 규칙 (정적 비용 제약)

- 차량·신호등 등 **동적 요소는 구현 범위에서 제외**한다.
- 횡단보도는 동적 통행 제어가 아니라, 로봇이 차도를 무단 횡단하지 않고 정해진 지점으로만 건너도록 유도하는 **정적 비용 제약**으로서 존재한다. 이는 로봇이 사람과 인도를 공유하며 차량보다 느리게 이동한다는 현실 전제를 경로 비용에 반영한 것이다.
- **연석(Curb):** 인도와 차도 경계에 높이차를 둔다.
  - 보행 로봇: 연석을 넘을 수 있어 무단 횡단을 "감수 가능".
  - 바퀴 로봇: 연석 높이에 따라 통행이 제약되어 사실상 횡단보도(연석이 낮아지는 지점)로 유도된다.
  - 연석 높이 수치는 실험 변수로 둔다(아래 7장 파라미터에 포함).

## 5. NavMesh 및 블록 단차 설계 원리

### 5.1 플랫폼별 NavMesh 분리
- 계단(`Path_Stair`)은 **바퀴 로봇 NavMesh에서 제외**하여 물리적으로 통행 불가로 만든다.
  - 사유: 바퀴 로봇이 계단 주행을 시도하다 끼이는 등의 예외 상황을 원천 차단. 현실의 로봇이 계단을 '갈 수 없는 경로'로 판단하는 것과 일치.
- 보행 로봇 NavMesh에는 계단이 포함된다.
- 따라서 NavMesh는 플랫폼(wheeled/legged)별로 별도 빌드한다. (구체적 빌드 방식은 Step 2-C에서 정의)

### 5.2 NavMesh 마진(Agent Radius) 동작
- NavMesh는 "에이전트(로봇)의 중심이 이동 가능한 영역"을 생성하므로, 블록 가장자리에서 Agent Radius(기본 `R=0.5m`)만큼 안쪽으로 들어간 곳까지만 NavMesh가 칠해진다.
- 결과적으로 절벽(다른 블록과 이어지지 않은 가장자리)에는 약 0.5m의 안전 마진이 남는다. 이는 로봇이 절벽 아래로 떨어지는 것을 방지하는 정상 동작이다.
- 서로 이어진 블록 사이(연속된 평면)는 마진 없이 NavMesh가 매끄럽게 연결된다.

### 5.3 블록 단차 설계 원리 (핵심)
NavMesh는 높이 변화의 형태에 따라 다르게 동작하며, 이를 블록 설계에 적극 활용한다:
- **완만한 경사면(경사 블록):** 평지-경사-평지가 연속적으로 이어지면 NavMesh가 하나로 연결된다. → 경사로(`Path_Ramp`)는 로봇이 통행 가능한 통로가 된다.
- **수직 단차(연석):** 인도(높이 10)와 차도(높이 8)처럼 2m 수직 단차가 있으면 NavMesh가 물리적으로 끊겨 분리된다. → 연석이 물리적 경계로 작동하여, 로봇이 차도로 임의 진입하지 못하고 횡단보도(단차가 없는 지점)로 유도된다.
- 즉, **"경사면=연결, 수직면=분리"**라는 NavMesh의 특성이 본 프로젝트의 설계 의도(경사로는 통하고 연석은 막힘)와 정확히 일치한다.

### 5.4 향후 확인 필요
- 경사 블록의 양 끝이 평지 블록과 만나는 지점에서 NavMesh가 실제로 끊김 없이 완벽하게 이어지는지는 경사 블록 검증 단계에서 실제 테스트로 확인해야 한다.
- Agent Radius를 플랫폼별(바퀴/보행)로 다르게 설정할 경우, 가장자리 마진 넓이와 좁은 틈새의 통행 가능 영역이 달라질 수 있으며, 이는 로봇 구현 단계(Step 3)에서 세부적으로 다룬다.

## 6. 비용 프로파일 (기준값)

### 6.1 비용 함수 (기존 정의)
```
Cost = distance_m × (1 + W_L×L + W_S×(1−S) + W_E×(1−E))
```
- L(부하율): 0~1, 높을수록 나쁨
- S(안정성): 0~1, 높을수록 좋음
- E(효율성): 0~1 기준, 내리막 등에서 1.0 초과 가능(상한 미설정 — 의도된 설계)
- 가중치: 바퀴 W_L=0.3, W_S=0.5, W_E=0.2 / 보행 W_L=0.2, W_S=0.3, W_E=0.5

### 6.2 지형별 기준 프로파일
아래 값은 "시뮬레이터가 해당 지형을 주행할 때 산출하도록 설계된 기준 기댓값"이며, DB에 하드코딩하는 고정값이 아니다. 실제 주행 시 노이즈가 더해진 측정값이 누적된다(6.3 참조).

| 지형 태그 | 바퀴 L/S/E | 보행 L/S/E | 비고 |
|---|---|---|---|
| Terrain_Flat | 0.10 / 0.95 / 0.95 | 0.20 / 0.90 / 0.70 | 평지 |
| Terrain_Slope | 0.50 / 0.60 / 0.50 | 0.40 / 0.75 / 0.60 | 경사·언덕 |
| Path_Ramp | 0.35 / 0.75 / 0.65 | 0.35 / 0.80 / 0.60 | 완만 우회로 |
| Path_Stair | 통행 불가 | 0.45 / 0.70 / 0.65 | 바퀴 NavMesh 제외 |
| Road_Sidewalk | 0.15 / 0.90 / 0.90 | 0.20 / 0.90 / 0.70 | 기본 주행로 |
| Road_Vehicle | 0.30 / 0.40 / 0.85 | 0.35 / 0.50 / 0.75 | 고비용 + 연석 제약 |
| Road_Crosswalk | 0.15 / 0.88 / 0.88 | 0.20 / 0.88 / 0.70 | 차도 건너는 통로 |
| Path_Tunnel | 0.20 / 0.85 / 0.85 | 0.25 / 0.85 / 0.65 | 선택. 평지에 준함 |

### 6.3 측정 노이즈 (경험 학습의 핵심)
- 시뮬레이터는 매 주행마다 기준값에 노이즈를 더한 측정값을 산출한다.
  ```
  측정값 = 기준값 + Uniform(−noise_range, +noise_range)   # 기본 noise_range = 0.05
  L, S 는 [0, 1] 범위로 클램프. E 는 상한 클램프하지 않음(내리막 1.0 초과 허용).
  ```
- 이유: 측정값이 매번 흔들려야 통계 누적(traversal_count 증가에 따른 평균 수렴)과 신뢰도(confidence_score) 개념이 실제 의미를 가진다. 고정값이면 데이터 누적이 무의미해진다.
- **노이즈 생성은 하나의 독립된 책임(모듈)으로 분리**하여, 향후 균등분포 → 정규분포(가우시안) 전환이나 변수 추가 시 해당 부분만 수정하도록 한다.
- 현 단계는 가장 단순한 균등 난수 버전으로 구현한다.

## 7. 파라미터 외부화 (공유 JSON)

모든 튜닝 파라미터는 코드에 하드코딩하지 않고 **공유 JSON 파일**로 외부화하여, 조정이 쉽고 언제든 기본값으로 되돌릴 수 있게 한다.

### 7.1 단일 원본 원칙
- 파라미터 원본은 **한 곳에만** 둔다(중복 복사본 금지). 중립 위치(예: 저장소 `config/`)에 두고 Unity와 Python이 각자 그 경로를 읽는다.
- 어느 한쪽(Unity/Python)의 소유가 아니라 공용 설정으로 취급한다.

### 7.2 파일 구성
- `config/cost_profiles.default.json` — 원본 기본값. **절대 수정하지 않음.**
- `config/cost_profiles.json` — 실사용 파일. default의 복사본으로 시작하며 튜닝 대상.
- "기본값으로 리셋" = default를 다시 복사.

### 7.3 JSON에 담을 섹션
1. 플랫폼 가중치: W_L/W_S/W_E (wheeled, legged)
2. 지형 기준 프로파일: 각 태그별 L/S/E (wheeled, legged)
3. 노이즈 폭: noise_range (기본 0.05)
4. 연석 높이: curb_height (실험 변수)

### 7.4 읽는 쪽
- **Python:** 가중치를 읽어 A* 비용 계산에 사용. (`cost_calculator`는 이미 weight_profile을 외부 주입받는 구조이므로, 주입 출처를 JSON으로 연결하면 됨. 알고리즘 로직 수정 불필요.)
- **Unity:** 지형 기준 프로파일 + 노이즈 폭 + 연석 높이를 읽어 측정값 생성에 사용.

### 7.5 조정 방식
- 현 단계는 **읽기 전용**으로 구현한다. JSON 수정 후 시뮬레이터 재시작 시 반영.
- Unity 인스펙터 실시간 슬라이더(양방향 쓰기)는 복잡도가 높으므로 현재 도입하지 않는다. 향후 필요 시 확장.

## 8. 백엔드 데이터 모델 연결 요약

- 거점 3종 + Waypoint → `nodes`(node_type='BASE') + `base_locations`(location_usage로 구분). 스키마 변경 불필요.
- 각 타일의 태그 → 해당 타일이 형성하는 엣지의 비용 프로파일 결정 → 주행 시 `map_edges`의 platform_stats(L/S/E)로 누적.
- 로봇이 자율 발견한 노드 → `discovered_nodes`(기존 구조 유지).

## 9. 미확정 / 후속 논의 항목

- 씬 덤프 도구의 상세 명세(수집 필드 스키마, 타일 인접 관계 포함 여부, 저장 경로·파일명·트리거 방식)는 별도 정의 예정.
- 플랫폼별 NavMesh 분리의 구체적 빌드 절차(Step 2-C).
