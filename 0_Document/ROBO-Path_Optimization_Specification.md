# ROBO-Path 복셀 시각화 및 렌더링 최적화 명세서

이 문서는 Unity 시뮬레이터에서 로봇이 탐색한 영역을 3D로 시각화하는 **복셀 기반 Fog of War 시스템**과, Mac Mini(M2 Pro, 16GB) 환경에서 상시 가동을 보장하기 위한 **렌더링·데이터 최적화 전략**을 정의합니다.

본 문서는 `ROBO-Path_Unity_Simulator_Architecture.md`의 3D 시각화 요구사항을 구현 레벨에서 구체화한 보조 명세이며, 기능 사양이 아닌 **성능·자료구조 설계**에 초점을 둡니다.

---

## 1. 3D 시각화 기본 방침

별도의 포인트 클라우드(.pcd) 파일을 생성하지 않는다. 대신 Unity 씬에 이미 존재하는 실제 3D 모델을 활용하되, 로봇이 Raycast로 실제 관측한 표면만 점진적으로 드러내는 **Fog of War(전장의 안개)** 방식을 사용한다.

- 로봇이 관측한 표면: 정상 렌더링
- 사물에 가려진 사각지대(뒷면 등): 미관측 처리 (드러내지 않음)
- 한 번도 탐색되지 않은 영역: 미탐색 처리 색상으로 채움

이 방식은 실제 로봇 LiDAR 센서를 1:1로 모사하지 않으며, "로봇이 어디를 보았는가"를 시각적으로 표현하는 것을 목적으로 한다.

---

## 2. 표면 복셀화 (Surface Voxelization)

복셀을 속이 꽉 찬 입체(Solid Voxel)로 구현하지 않는다. 로봇이 Raycast로 관측 가능한 것은 사물의 **드러난 표면**뿐이므로, 표면에 해당하는 복셀만 생성·관리한다.

- Ray가 표면에 히트하는 순간, 해당 히트 지점이 속한 표면 복셀만 "관측됨"으로 활성화한다.
- 건물 내부, 지면 아래 등 관측 불가능한 내부 볼륨은 복셀화하지 않는다.
- 이를 통해 Solid 방식 대비 복셀 수를 대폭 절감한다 (건물 한 채 기준 수천 개 → 표면 수백 개 수준).

---

## 3. 동적 복셀 해상도 (Octree 기반)

복셀 해상도를 고정하지 않고, **Octree(옥트리) 자료구조**를 사용하여 영역 특성에 따라 동적으로 분할한다.

### 3.1 분할 원칙
- 넓고 평탄한 면(운동장, 도로, 광장): 큰 복셀로 병합하여 데이터 절감
- 작고 복잡한 사물(계단, 화단, 표지판, 좁은 통로): 작은 복셀로 세분화하여 디테일 보존

### 3.2 해상도 범위
- **최소 복셀 크기:** 0.1m³ (가장 세밀한 단위)
- **최대 복셀 크기:** 설정값으로 제어 (초기 권장 2m³)
- Octree는 큰 노드에서 시작하여 디테일이 필요한 영역만 8분할을 재귀적으로 반복한다.

### 3.3 동적 조절
- 복셀 최소/최대 크기 및 Octree 최대 깊이는 `.env` 또는 설정 파일로 관리한다.
- Mac Mini 성능 측정 결과에 따라 런타임 조정이 가능해야 하며, 값을 코드에 하드코딩하지 않는다.

---

## 4. 2.5D 높이 처리

야외 캠퍼스 환경은 높이 변화가 지형을 따라 비교적 완만하므로, 전 공간을 3D 정육면체 복셀로 꽉 채우지 않는다.

- 바닥 격자(2D) + 각 격자의 높이값(Height) 조합인 **2.5D 방식**을 기본으로 한다.
- 계단, 다층 구조물 등 수직 디테일이 필요한 구역에 한해 Octree로 수직 분할을 허용한다.

### 4.1 경계 처리 검증 (필수)
2.5D 바닥 구조와 3D Octree 수직 구조가 만나는 경계(예: 언덕 지형과 건물 계단이 이어지는 지점)에서 두 자료구조 병합 시 버그가 발생하기 쉽다.

- 경계 판별 및 병합 로직은 구현 초기에 우선적으로 작성한다.
- 이 로직은 EditMode 테스트로 **가장 먼저 검증**한다. (예: 2.5D 격자와 Octree 노드가 인접한 경우 중복/누락 복셀이 없는지 확인하는 테스트)

## 5. 탐색 시점 기반 명암 (Temporal Fading)

각 복셀에 마지막으로 관측된 시각(timestamp)을 기록하고, 현재 시각과의 경과 시간에 따라 밝기를 조절한다.

- 최근 관측 영역: 밝게 표시
- 오래 전 관측 영역: 어둡게 표시
- 동일 영역이 재관측되면 timestamp를 갱신하여 다시 밝아진다.

### 5.1 CPU/GPU 역할 분리 (필수)
밝기 감쇠를 CPU(C#)에서 매 프레임 계산하여 복셀 색상을 갱신하면 CPU 부하가 급격히 증가한다. 따라서 역할을 아래와 같이 분리한다.

- **CPU(데이터 레이어):** 각 복셀에 '최초/마지막 관측 timestamp'만 부여하고 저장한다. 이 작업은 헤드리스 모드에서도 계속 수행된다.
- **GPU(렌더링 레이어):** 실제 밝기 감쇠 효과는 커스텀 셰이더가 담당한다. 셰이더가 현재 시간(`_Time`)과 복셀의 관측 timestamp를 비교하여 동적으로 밝기를 렌더링한다.
- **헤드리스 주의:** 헤드리스 모드에서는 셰이더가 동작하지 않으므로 밝기 렌더링은 수행되지 않는다. 단, timestamp 누적은 CPU 데이터 레이어에서 계속 이루어지므로, GUI 모드로 전환하면 누적된 데이터를 기반으로 명암이 정상 표시된다.

> **데이터 연계:** 복셀의 마지막 관측 시각은 `discovered_nodes`의 신뢰도(`confidence_score`) 개념과 연계 가능하다. 오래 관측되지 않은 영역일수록 신뢰도가 낮다고 해석할 수 있다.

---

## 6. 미탐색 영역 표시 옵션

한 번도 탐색되지 않은 영역의 표시 방식을 사용자가 선택할 수 있도록 한다.

- **검정(Black):** 기본값
- **투명(Transparent):** 미탐색 영역을 비워 둠
- **하양(White):** 미탐색 영역을 흰색으로 채움

이 옵션은 Streamlit 관제 UI에서 토글로 제공하며, 선택값을 Unity 뷰어에 전달하여 셰이더 파라미터로 반영한다.

---

## 7. 렌더링 부하 최적화

### 7.1 거리 기반 컬링 (Culling)
- 매 프레임 맵 전체를 렌더링하지 않는다.
- 로봇(또는 카메라) 위치 기준 일정 반경 내의 복셀/청크만 로드·렌더링한다.
- Unity의 Occlusion Culling 및 LOD(Level of Detail)를 병행 활용한다.

### 7.2 청크(Chunk) 단위 관리
- 맵을 일정 크기의 구역(청크)으로 분할한다.
- 탐색된 청크만 메모리에 로드하고, 멀리 떨어진 청크는 언로드한다.
- 청크 단위로 복셀 데이터를 파일에 저장/로드하여 메모리 사용량을 제어한다.

### 7.3 헤드리스 모드 처리
- `-batchmode -nographics` 헤드리스 모드에서는 렌더링 파이프라인을 비활성화한다.
- 복셀 관측 상태(어느 복셀이 관측되었는가)는 헤드리스에서도 계속 누적·기록한다.
- 즉, 렌더링은 GUI 모드에서만 수행하되, 데이터 갱신은 모드와 무관하게 동작한다.

---

## 8. 복셀 데이터 저장 전략

### 8.1 하이브리드 스토리지
복셀 관측 데이터는 용량이 커질 수 있으므로 Supabase(클라우드 DB)에 직접 저장하지 않는다.

- **파일 본체:** 라즈베리파이 1TB SSD에 저장 (FastAPI 스토리지 서버 경유)
- **경로 인덱스:** Supabase `discovered_nodes.pcd_file_url` 필드에 파일 경로만 저장

### 8.2 `pcd_file_url` 필드 의미 재정의
- **기존 의미:** 발견 노드의 3D 포인트 클라우드(.pcd) 파일 경로
- **변경 의미:** 발견 노드의 **탐색 복셀 데이터 파일 경로** (Octree 직렬화 결과)
- 필드명은 스키마 변경 최소화를 위해 유지하되, 의미만 본 문서에서 재정의한다.

### 8.3 직렬화 포맷
- Octree 구조를 직렬화하여 청크 단위 파일로 저장한다.
- **바이너리 포맷을 우선 사용한다.** 대규모 Octree 구조를 JSON으로 직렬화/역직렬화할 경우 막대한 메모리 할당이 발생하여 가비지 컬렉터(GC) 스파이크로 인한 프레임 드랍이 발생할 수 있다.
- 초기 구현은 Unity 내장 바이너리 직렬화로 시작하고, 병목이 확인되면 MessagePack 등 고성능 바이너리 라이브러리로 교체한다. 외부 라이브러리 의존성은 병목 확인 후 도입하여 초기 복잡도를 낮춘다.
- 각 복셀 레코드는 위치, 크기(Octree 깊이), 마지막 관측 timestamp를 포함한다.

---

## 9. 우선순위 및 단계적 구현

본 시각화 시스템은 프로젝트 **필수 기능**이다. 단, 아래 순서로 점진 구현하여 각 단계가 독립적으로 동작하도록 한다.

1. **표면 복셀화 + 고정 해상도(2m³):** 로봇이 본 표면만 단순 복셀로 표시
2. **Octree 동적 해상도:** 영역별 복셀 크기 가변 적용 (최소 0.1m³)
3. **컬링 + 청크 관리:** 렌더링 부하 최적화
4. **Temporal Fading + 미탐색 색상 옵션:** 시각적 완성도

각 단계는 이전 단계가 검증된 후 진행하며, 시간 제약 시 1~2단계까지만으로도 핵심 시연이 가능하도록 설계한다.

## 10. 하드웨어 전제

- **기본 운영 환경:** Mac Mini (M2 Pro, 10코어 CPU / 16코어 GPU, 16GB RAM, 512GB)
- 위 최적화(표면 복셀화 + Octree + 컬링 + 청크 + 2.5D)를 모두 적용하면 500m × 500m 규모를 Mac Mini에서 상시 가동 가능한 것으로 판단한다.
- 만약 성능이 부족할 경우, Ubuntu 22 기반 워크스테이션(RTX 5090)으로 시뮬레이터 구동 환경을 이전한다. 이 경우에도 코드 변경 없이 실행 환경만 교체한다.

---

## 11. 향후 확장 (현 단계 미적용)

아래 항목은 기술적으로 유효하나, 현 단계(로봇 1~2대, 단기 마감)에서는 학습 비용 및 복잡도 대비 효용이 낮아 적용하지 않는다. 다중 로봇 운용 등으로 부하가 증가할 때 도입을 검토한다.

### 11.1 Job System 기반 병렬 Raycast
- 현재는 단일 로봇 기준 36개 Raycast를 메인 스레드에서 처리한다.
- 로봇 수가 늘거나 스캔 주기가 짧아져 메인 스레드 물리 연산이 병목이 될 경우, Unity의 Job System과 `RaycastCommand`를 활용하여 M2 Pro의 멀티 코어에서 Raycast를 병렬·비동기 처리한다.
- 1차 구현은 일반 Raycast로 진행하며, 병목이 실측으로 확인된 후에만 전환한다.
